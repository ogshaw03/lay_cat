# LayCAT セキュリティ・運営視点メモ

将来的にサブスク展開や複数社利用する際の、運営から見えるデータ範囲と対応方針の検討メモ。
2026-07-21 時点の議論内容。実装未着手。

---

## 現状の運営視点で見える範囲

Cloudflare 経由で運営（LayCAT 管理者）が触れるものの整理：

| 見えるもの | 範囲 | 対策の有無 |
|---|---|---|
| 監査ログ（誰がいつ何を触ったか） | すべて見える | 今後実装しても運営には見える |
| R2 バケット内のファイル一覧 | すべて見える | Cloudflare ダッシュボードで見れる |
| プロジェクトメタデータ（laycat.project.json） | 平文なら丸見え | プロジェクト暗号化機能で対応可能 |
| 動画・画像本体 | ダウンロード可能 | 暗号化対象外・見えてしまう |
| コメント本文 | 平文なら丸見え | 暗号化で対応可能 |
| ユーザーのメール | 見える | Firebase Auth の性質上避けられない |

**LayCAT に限らず、全ての SaaS で同じ構造。** ShotGrid（Autodesk）・Frame.io（Adobe）・ftrack も、運営会社は技術的に顧客データにアクセス可能。

## 「NDA だけで十分」なケース

- 顧客企業間で信頼関係がある
- アニメ制作・映像制作など、業界内で標準的な情報（原則公開後には見せる作品）
- 1〜数社との個別契約ベースの関係
- 個人情報を大量に扱わない

**必要な整備**：
1. 秘密保持契約（NDA）
2. プライバシーポリシー（データ扱いの明示）
3. 利用規約（責任範囲の明示）

映像制作業界では ShotGrid・Frame.io レベルでも「NDA + サービス契約」で運用されている。

---

## 「NDA だけでは不十分」なケース

| 状況 | 追加で必要なもの |
|---|---|
| 未公開作品（劇場版・大作アニメの本編） | クライアント側暗号化 or 専用インスタンス |
| 個人情報保護法（APPI）対象データを扱う | データ処理契約（DPA）・削除ポリシー・第三者監査 |
| 上場企業・大手ゲーム会社との取引 | SOC 2 認証・ペネトレーションテスト報告書 |
| 国防・政府案件 | 該当セキュリティ基準クリア（要相談） |
| 医療・金融データ | 業界別コンプライアンス（HIPAA/PCI DSS 等） |
| 海外の企業（特に EU）と取引 | GDPR 準拠・DPA・EU 域内保存要件 |

## LayCAT の強み：プロジェクト暗号化機能

LayCAT は既にクライアント側暗号化（PBKDF2 → AES-GCM）を実装済み。パスワードを設定したプロジェクトは：

- `laycat.project.json` はパスワードで暗号化されて R2 に保存
- **運営（管理者）でも復号できない**（パスワードを知らないので）
- コメント・アノテーション・ステータス全部が暗号化対象

**現状の制限**：
- 動画・画像ファイル本体は暗号化対象外（技術的困難）
- 動画は依然として運営が R2 ダッシュボードで見れる

## 実務上の推奨 3 段階構成

### Tier 1：一般利用（NDA + 標準サービス）
- NDA 締結
- プライバシーポリシー・利用規約明示
- 「運営はサービス運用上、データにアクセス可能ですが、内部規定により…」と明記
- 映像制作業界の 9 割はこれでカバー

### Tier 2：機密案件（+ プロジェクト暗号化）
- Tier 1 + プロジェクトパスワード必須
- クライアントに「パスワードは絶対に運営に教えないでください」と明示
- 動画本体もクライアントで事前に別暗号化（Zip パスワード等）してからアップ

### Tier 3：ハイエンド（+ 専用環境）
- Tier 2 + 別 Cloudflare アカウント・別 R2 バケット・専用 Worker
- クライアント側で Cloudflare を自社契約 → LayCAT は「単なる HTML」として使う
- 運営は運用のみ担当、データアクセス権なし

---

## 個人・小規模運営 LayCAT への推奨アクション

1. まず **NDA + 利用規約 + プライバシーポリシー**の 3 点セットを整備
2. **プライバシーポリシーに「運営がデータにアクセスできる範囲と目的」を明示**
   - 例文：「運営は障害対応・不正利用調査の目的で技術的にデータへアクセスすることが可能ですが、業務目的以外での閲覧・利用は行いません」
3. **機密案件は「プロジェクト暗号化」機能を必須にする案内**
4. 監査ログ機能を「運営の適切な運用の証拠」として説明できるようにする
   - 「誰も見ていない」ではなく「見た記録が全部残っている」の方が信頼される

## 参考：Frame.io の運用モデル（同業他社）

- 運営はアクセス可能を明示
- ISO 27001・SOC 2 認証で内部統制を担保
- クライアント側暗号化は Enterprise プランでのみ提供

**個人運営 LayCAT で参考にすべきポイント**：
- Frame.io レベルの認証はまだ不要（費用が数百万円かかる）
- **NDA + 利用規約 + プライバシーポリシー + プロジェクト暗号化オプション** で十分な信頼構築が可能

## 監査ログの access-console 統合について

このセキュリティ議論を踏まえて、監査ログ機能自体は「不安要素」ではなく「信頼構築の材料」と位置付ける。

- 「誰がいつ何を触ったか全部記録している」と説明できる
- 事故発生時に「運営側の不正閲覧ではない」と証明できる
- クライアントからの調査依頼にも応えられる

**実装時の配慮**：
- 運営のみアクセス可能な画面に配置
- IP や UA も記録（不正調査用）
- 監査ログ自体の閲覧履歴も記録（メタ監査）

詳細な実装計画は `UPDATE_LOG.md` の TODO セクション参照。

## R2 プロジェクト階層のフラット構造問題（**2026-07-24 対応済み**）

**採用案 C（Firestore メンバーシップ表）を実装済み**。以下参照：

- Worker が `_access.json` 変更時に `laycatMemberships/{emailKey}/projects/{pid}` を自動同期
- `/api/r2/mine` は Firestore メンバーシップ表を LIST するだけに変更（R2 バケット全 LIST を廃止）
- 運営が明示的に `?bucketScan=1` を付けると旧経路（バケット全 LIST）にフォールバック可能
- 初回移行は運営が access-console の「🔄 R2 メンバーシップ再構築」ボタン、または `POST /api/r2/rebuild-memberships` を 1 回叩く
- Firestore rules で `laycatMemberships/{key}/projects/{pid}` の読取を各ユーザーの自分の emailKey のみに制限、書込は Service Account 経由のみ

**残る運用上の注意**：
- Cloudflare ダッシュボードから R2 バケット LIST を直接見た場合、依然としてプロジェクト構造は見える（Cloudflare 管理者権限を持つ運営のみ）。これはインフラ運営者が閲覧可能というだけで、通常ユーザーの経路からは見えない。
- Firestore 管理コンソールから `laycatMemberships` を直接見た場合、全ユーザーのプロジェクト参加状況が見える（Google Cloud 管理者権限を持つ運営のみ）。同上。

## （旧・記録用）R2 プロジェクト階層のフラット構造問題

**現状の問題：**
現在の R2 バケット構造は `projects/<projectId>/...` と平置きになっており、以下の課題がある。

| 課題 | 影響 |
|---|---|
| 運営（Worker 運用者）がバケット LIST で全プロジェクト ID・owner・members を見れる | プライバシー |
| `/api/r2/mine` が毎回全プロジェクトの `_access.json` を読む | パフォーマンス（プロジェクト数に比例） |
| Cloudflare の LIST/GET リクエスト課金が積み上がる | コスト（数千規模で気にする程度） |

**採用予定の対策：案 C（Firestore メンバーシップ表）**

```
Firestore:
  laycatMemberships/{sha256(email)}/projects/{pid}: { role, joinedAt, ... }
```

- 既存の R2 データ構造は変えない（移行不要）
- `/api/r2/mine` は Firestore のメンバーシップ表を LIST するだけ
- 運営はアクセス管理コンソール経由でしか閲覧しない前提
- パフォーマンス最良（O(自分のプロジェクト数)）
- Firestore 無料枠内でコスト増なし

**実装コスト：** 2〜3 日

**同期処理：**
- プロジェクト作成時／メンバー追加時：Firestore に該当ユーザーのメンバーシップを追加
- メンバー削除時：該当エントリを削除
- プロジェクト削除時：全メンバーのエントリを削除
- 既存プロジェクトの初回移行：Worker で全プロジェクトを走査してインデックス生成

**却下した案：**
- 案 A（R2 上に `indexes/users/<hash>/projects/<pid>`）：R2 上のインデックス管理が煩雑で二重管理感が強い
- 案 B（`users/<hash>/projects/<pid>/...` に物理隔離）：共有プロジェクトの扱いが複雑、既存データ移行が重い

**着手時期：** サブスク本格運用前の実装項目リストに追加。優先度は中〜高。
